Step 2.5 — Hybrid AI Discovery

URL-First: AI Discovers Scenarios from Live Pages

Provide only a URL. The platform crawls the live page, extracts every form field, button, nav link, and heading — then uses AI to infer all testable scenarios. Scenarios you wrote in Excel are preserved and AI adds on top of them.

① PageAnalyzer — Playwright DOM Extraction
Headings: "Welcome", "Login", "Create Account"
Forms: username field, password field
Buttons: "Sign In", "Forgot Password", "Register"
Nav links: Home, Products, Cart, Signup/Login
↓ ScenarioInferenceEngine (LLM) ↓
② AI-Discovered Scenarios
✦ Verify "Forgot Password" link navigates correctly
✦ Test "Register" button leads to registration form
✦ Verify nav links visible and functional
✦ Test login form with empty fields shows validation
✦ Verify page heading renders on load
  • 1
    Playwright launches headless Chromium, navigates to the URL
  • 2
    DOM is parsed: forms, buttons, headings, nav links extracted into PageContext
  • 3
    LLM receives PageContext + your existing Excel scenarios → infers what else is testable
  • 4
    New scenarios appended as AI-Discovered requirements — your Excel rows untouched
  • 5
    Results cached in knowledge-base/{page}-scenarios.json — no re-crawl on next run

Self-Healing Locator — Live Demo

ParaBank updated its login button. The old selector breaks. The healing engine finds the correct one automatically.

"#loginForm .btn-primary"
// TimeoutError: waiting for selector

✅ HEALED SELECTOR
{

  healedLocator:   "input[value='Log In']",
  confidence:      94,
  reasoning: "Login button found in KB
    under selectors.loginButton"
}
import { SelfHealingLocatorEngine }
  from './SelfHealingLocatorEngine.js';
import { KnowledgeBaseService }
  from '../../knowledge-base/...';

const engine = new SelfHealingLocatorEngine(
  llmProvider
);

const kb = new KnowledgeBaseService()
  .load('login-page');

const result = await engine.heal(
  {
    failedLocator: '#loginForm .btn-primary',
    pageName:      'login-page'
  },
  kb
);

console.log(result.healedLocator);
// → "input[value='Log In']"
console.log(result.confidence);
// → 94

Enterprise Optimization · Performance

Four Layers of Performance Engineering

⚡

LLM Response Cache

Every LLM call is SHA-256 hashed. Identical prompts return instantly from disk — zero API calls, zero tokens consumed.

// CachingLLMProvider wraps any provider
// sha256(version + prompt) → .llm-cache/
📦 LLM cache: 47 hits / 5 misses
   (90% cached — saved ~38 API calls)
Files: llm/src/providers/CachingLLMProvider.ts
🔀

8× Concurrency Pool

All requirements expand in parallel (Step 4). All test cases within a page generate in parallel (Step 5). Wall-clock time = slowest single item.

// pMap — concurrency-capped parallel map
const expanded = await pMap(
  requirements,
  req => expander.expand(req, kb),
  8  // 8 parallel LLM calls
);
Files: ai/src/utils/concurrency.ts
🎯

Rule-Based Action Classifier

~90% of test steps are classified by keyword regex patterns — zero LLM calls. Only novel steps the rules don't recognise fall back to AI.

// "Click the Login button"  → click (rule)
// "Enter valid username"    → fill  (rule)
// "Navigate to login page"  → goto  (rule)
// "Verify error is visible" → noop  (rule)
// "Do something unusual"    → AI    (fallback)
Files: ai/src/action-model/RuleBasedActionModelGenerator.ts
📋

Spec File Manifest

Generated spec files are content-hashed. If test cases for a page didn't change, spec generation is skipped entirely — not even the TypeScript renderer runs.

// .llm-cache/spec-manifest.json
// hash(allTestCases) matches stored hash
// AND all spec files still exist on disk
